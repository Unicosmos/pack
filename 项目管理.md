## 二、项目问题诊断

### 1. 文件组织结构问题

| 问题             | 具体表现                                                         | 影响               |
| ---------------- | ---------------------------------------------------------------- | ------------------ |
| **模块边界模糊** | `utils/box_detector.py` 与 `YOLO/pack_predict.py` 功能重叠       | 代码重复，维护困难 |
| **数据目录分散** | `sku_library/`、`SKU/sku_auged/`、`SKU/sku_output/` 多处数据存储 | 数据管理混乱       |
| **配置文件散落** | 配置分散在 `YOLO/configs/`、`web/backend/config.py`              | 配置管理复杂       |

### 2. 命名规范问题

| 问题               | 具体表现                                                     | 建议                                                    |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------- |
| **命名风格不一致** | `sku_augmentation.py` vs `box_detector.py`（下划线 vs 小写） | 统一使用 snake_case                                     |
| **文件命名过长**   | `sku_model_trainer.py`、`sku_augmentation.py`                | 可简化为 `trainer.py`、`augmentation.py`（在SKU模块内） |
| **目录命名不规范** | `sku_library/`、`SKU/` 大小写不一致                          | 统一小写                                                |

### 3. 依赖关系问题

| 问题             | 具体表现                                                         | 影响         |
| ---------------- | ---------------------------------------------------------------- | ------------ |
| **循环依赖风险** | `web/backend/core/matcher.py` 依赖 `../SKU/feature_extractor.py` | 模块耦合度高 |
| **隐式依赖**     | `main.py` 直接访问 `../sku_library/` 硬编码路径                  | 可移植性差   |
| **环境依赖**     | 未明确指定依赖版本（如 `ultralytics`、`torch`）                  | 环境复现困难 |

### 4. 版本控制问题

| 问题               | 具体表现                                 | 影响     |
| ------------------ | ---------------------------------------- | -------- |
| **大文件未排除**   | `sku_library/images/` 可能被纳入版本控制 | 仓库膨胀 |
| **训练结果未排除** | `YOLO/runs/` 可能被纳入版本控制          | 仓库膨胀 |
| **缺乏CHANGELOG**  | 无变更记录                               | 追溯困难 |

### 5. 文档问题

| 问题             | 具体表现                                           | 影响         |
| ---------------- | -------------------------------------------------- | ------------ |
| **文档分散**     | 文档分散在 `md/`、`SKU/README.md`、`web/README.md` | 查找困难     |
| **缺乏API文档**  | 后端API仅依赖FastAPI自动生成                       | 接口使用不便 |
| **代码注释不足** | 核心算法缺少注释说明                               | 维护困难     |

### 2. 命名规范统一

**制定命名规范**：

| 类型      | 规则             | 示例                |
| --------- | ---------------- | ------------------- |
| 包/模块   | 小写，下划线分隔 | `sku.extractor`     |
| 类        | PascalCase       | `FeatureExtractor`  |
| 函数/方法 | snake_case       | `extract_feature()` |
| 变量      | snake_case       | `sku_features`      |
| 常量      | UPPER_SNAKE_CASE | `FEATURE_DIM = 384` |
| 数据目录  | 小写，连字符分隔 | `sku-library/`      |

**实施步骤**：
1. 在 `docs/` 中创建 `CODING_STYLE.md` 文档
2. 逐步重构现有代码
3. 使用 lint 工具（如 `flake8`）检查

---

### 3. 模块划分调整

**建议的模块职责边界**：

| 模块                    | 职责             | 边界                   |
| ----------------------- | ---------------- | ---------------------- |
| `sku.extractor`         | 特征提取器封装   | 仅负责特征提取         |
| `sku.matcher`           | SKU匹配逻辑      | 仅负责相似度计算和匹配 |
| `detector.yolo`         | YOLO检测封装     | 仅负责目标检测         |
| `detector.box_detector` | 箱体检测业务逻辑 | 调用YOLO并处理结果     |
| `utils.image_utils`     | 通用图像处理函数 | 无业务逻辑             |

**解耦策略**：
- 通过接口抽象降低模块耦合
- 使用依赖注入方式传递依赖
- 定义清晰的模块接口契约

## 五、总结

当前项目已经实现了完整的箱货检测与SKU匹配功能，但在**可维护性**、**可读性**和**开发效率**方面仍有提升空间。通过实施上述优化建议，可以：

1. **降低维护成本**：清晰的模块划分和命名规范
2. **提升开发效率**：完善的文档和自动化测试
3. **增强可扩展性**：解耦的架构设计
4. **保障代码质量**：CI/CD流程和代码审查

建议按照优先级逐步实施，先解决低风险、高收益的问题，再逐步推进架构重构工作。
