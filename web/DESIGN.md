# Pack Web 产品设计文档

> 面向物流仓储场景的箱货检测与SKU匹配Web应用
>
> 基于YOLO-cheap轻量化检测 + OML度量学习SKU匹配

---

## 1. 项目概述

### 1.1 产品定位

Pack Web 是一款面向物流仓储场景的**箱货检测与SKU匹配Web应用**。用户上传包含地堆箱货的图片，系统自动完成两阶段识别流程：先通过YOLO检测模型定位箱体位置，再通过度量学习特征匹配识别具体SKU，实现仓储货物的快速识别和数字化管理。

### 1.2 核心功能

| 功能模块 | 描述 | 技术实现 |
|----------|------|----------|
| 箱货检测 | 自动定位图像中的箱体位置，输出检测框+分割掩码 | YOLOv8-seg-cheap + 边界感知损失 |
| SKU匹配 | 提取箱体视觉特征，与SKU特征库进行相似度匹配 | ViT-S16 DINO + 余弦相似度 + Ratio Test |
| 结果展示 | 可视化检测框、匹配状态颜色编码、裁剪图预览、统计信息 | PIL绘图 + Base64传输 |
| SKU库管理 | SKU图片采集、增强、特征入库、库浏览 | 固定方案增强 + NumPy特征矩阵 |
| 参数调节 | 检测置信度阈值、匹配相似度阈值由后端配置管理，不暴露给前端用户 | 后端配置文件管理 |

### 1.3 技术选型

| 层级 | 技术方案 | 选择理由 | 不选替代方案及原因 |
|------|----------|----------|-------------------|
| 后端框架 | FastAPI | 自动生成Swagger API文档（/docs），答辩演示时可直接展示接口设计；基于类型注解的参数自动校验，减少手动校验代码；原生支持WebSocket，为后续实时进度推送留扩展空间 | 不选Flask：需额外安装flask-swagger等扩展才能实现自动API文档，参数校验需手写；Flask基于WSGI不支持异步，若后续需WebSocket需加扩展 |
| 深度学习 | Ultralytics YOLO + PyTorch | 检测模型训练与推理的标准工具链 | 不选MMDetection：Ultralytics API更简洁，毕设阶段快速迭代优先 |
| 特征匹配 | OML度量学习框架 + ViT-S16 DINO | 自监督预训练泛化性强，384维特征区分度高 | 不选ResNet50+分类头：SKU品类会变化，度量学习不需要重新训练分类层<br>不选CLIP：CLIP偏语义理解，SKU视觉差异细微需要像素级特征 |
| 前端 | Vue.js | 前端交互状态多（上传/检测中/结果展示/空状态/错误状态），Vue响应式数据绑定比手动操作DOM更清晰可维护；组件化结构便于答辩演示时快速定位和修改 | 不选纯HTML+Vanilla JS：状态管理靠手动操作DOM，容易出错且难维护；不选React：毕设阶段Vue学习曲线更平缓，中文生态更友好 |
| 数据存储 | NumPy + JSON | 训练/建库/匹配三阶段分离：训练产出微调模型权重，建库用模型提取特征存npy，匹配阶段只做向量计算，无需数据库依赖 | 不选SQLite/数据库：35 SKU特征矩阵仅~100KB，numpy矩阵运算直接高效；训练阶段不产生特征文件，避免混淆 |
| 部署模式 | Uvicorn托管前端 | Uvicorn是ASGI服务器，负责接收HTTP请求并转发给FastAPI处理；类比Nginx之于静态网站，Uvicorn之于FastAPI就是"让框架跑起来的服务器程序"；单机演示场景一条命令启动 | 不选Nginx反代：单机演示场景减少部署步骤；不选Gunicorn：Gunicorn是WSGI服务器，无法配合FastAPI的ASGI架构 |

## 3. 核心使用场景

### 3.1 场景一：仓库盘点

**场景描述**：仓库管理员在仓储区域拍摄地堆箱货照片，上传系统，快速识别箱体数量和对应SKU，与台账对比完成盘点。

**典型操作流程**：
1. 管理员在地堆前拍摄包含多个箱体的照片
2. 登录系统，上传照片
3. 系统自动检测并匹配SKU
4. 管理员核对识别结果与台账，对差异项进行复核

**关键需求**：
- **低置信结果需要人工复核入口**：当系统返回 `low_conf` 状态时，结果区域应高亮显示该箱体，并提供"标记为待复核"按钮，管理员可手动确认或修改SKU编号
- 检测数量统计应与台账品类数量对照展示
- 支持多次上传逐步完成大区域盘点

**用户价值**：原本需要逐一扫描每个箱体条码的盘点工作，现在可以快速批量识别，显著提升盘点效率。

### 3.2 场景二：入库核验

**场景描述**：货物入库时，拍摄整箱货物照片，系统检测箱体并匹配SKU，核对入库品类是否与预期一致。

**典型操作流程**：
1. 货车到货，拍摄整箱货物照片
2. 上传系统，设置预期入库SKU列表
3. 系统检测并匹配，返回每个箱体的识别结果
4. 系统标记与预期不符的箱体，管理员确认处理

**关键需求**：
- **匹配失败的箱体需要高亮标记，不支持跳过**：当检测到 `unmatched` 状态的箱体时，必须强制人工介入，系统禁止忽略该结果
- 结果列表按匹配状态分组展示，未匹配项置顶
- 匹配结果与预期SKU列表对照，差异项用红色标识

**用户价值**：入库核验从"逐件扫码"变为"整图批量识别"，异常品自动高亮，大幅减少人工核对工作量。

### 3.3 场景三：SKU库维护

**场景描述**：管理员采集新SKU图片，通过固定增强方案生成训练样本，提取特征入库，并验证特征区分度。

**典型操作流程**：
1. 采集新SKU的标准图片（正面清晰、光照均匀）
2. 通过增强工具生成5种变体图（旋转、亮度变化、模糊）
3. 提取所有图片的特征向量，构建新SKU的特征模板
4. 系统计算新SKU与现有SKU的特征距离，验证区分度
5. 确认区分度合格后入库

**关键需求**：
- **入库后需验证特征区分度**：新SKU入库时，系统自动计算与最近邻SKU的余弦相似度，若相似度 > 0.85 则提示"特征过于相似，建议复核"
- 支持查看新SKU与候选混淆SKU的对比图
- 特征区分度报告可导出存档

**用户价值**：新SKU入库流程规范化，避免因特征区分度不足导致的匹配错误，保障系统长期维护质量。

### 4.2 前端交互状态机

#### 4.2.1 状态定义

| 状态 | 说明 | UI表现 | 约束条件 |
|------|------|--------|----------|
| `IDLE` | 初始状态 | 上传区域可点击，显示拖拽提示 | 只能执行"上传"操作 |
| `UPLOADED` | 图片已上传，待检测 | 预览图+文件名+参数滑块+检测按钮 | 只能执行"检测"或"移除" |
| `PROCESSING` | 检测处理中 | 按钮显示Loading动画，上传区域禁用 | 禁止任何交互，等待响应 |
| `SUCCESS` | 检测成功 | 统计卡片+结果图+详情列表 | 可执行"重新检测"或"移除" |
| `ERROR` | 检测失败 | 错误提示+重试按钮 | 只能执行"重试"或"移除" |
| `SYSTEM_INIT` | 系统初始化中 | 页面顶部红色横幅+禁用所有操作 | 持续至系统就绪 |

#### 4.2.3 各状态UI表现详细说明

**IDLE 状态**：
- 上传区域：显示虚线边框 + 上传图标 + "点击或拖拽上传图片" 文字
- 拖拽悬停时：边框变为蓝色，背景变为浅蓝色
- 按钮禁用

## 5.5 SKU多面匹配策略

### 问题背景

一个SKU可能有多个面（正面、侧面等），不同面的视觉特征差异大。如果训练时强制把同SKU多面拉近，会导致特征空间"折叠"，压缩不同SKU之间的距离，影响区分度。尤其是同品牌不同口味的SKU，正面设计语言高度统一，强行多面拉近会削弱模型对口味差异的区分能力。

### 设计决策：以面为Label训练，两级检索匹配

**核心原则**：一个面（face）作为一个训练label，一个SKU可以有多个label。训练时模型学习区分每个label，匹配时先按label找到top5候选，再映射到top1 SKU。

| 阶段 | 策略 | 目的 |
|------|------|------|
| **训练阶段** | 每个面作为独立label（如 `000001_front`、`000001_side_a`），OML Triplet Loss按label拉近同类、推远异类 | 模型学习细粒度区分能力，不同面不会被强制折叠到一起 |
| **建库阶段** | 所有面的图片提取特征存入矩阵，用CSV记录图片→label→SKU的映射关系 | 覆盖SKU的多种样貌，同时保留label层级索引 |
| **匹配阶段** | 先按label找top5最相似标签，再从top5中映射到top1 SKU，取top1 SKU | label级检索更精准，SKU级聚合保证最终结果正确 |

### SKU库数据格式

**sku_library.csv**：
```csv
image_name,sku_id,label,sku_name
000001/front.jpg,000001,000001_front,康师傅红烧牛肉面
000001/aug_left_front.jpg,000001,000001_front,康师傅红烧牛肉面
000001/aug_right_front.jpg,000001,000001_front,康师傅红烧牛肉面
000001/aug_dark_front.jpg,000001,000001_front,康师傅红烧牛肉面
000001/aug_bright_front.jpg,000001,000001_front,康师傅红烧牛肉面
000001/aug_lowq_front.jpg,000001,000001_front,康师傅红烧牛肉面
000001/side_a.jpg,000001,000001_side_a,康师傅红烧牛肉面
000001/aug_left_side_a.jpg,000001,000001_side_a,康师傅红烧牛肉面
...
000002/front.jpg,000002,000002_front,康师傅香辣牛肉面
...
```

**字段说明**：
- `image_name`：图片文件名，对应 `sku_library/images/` 下的文件
- `sku_id`：SKU编号，6位数字
- `label`：训练标签，格式为 `{sku_id}_{face_name}`，是训练和检索的基本单位
- `sku_name`：SKU名称，供前端展示

**为什么用CSV而不用JSON**：
- CSV天然是行记录结构，一行就是一张图的完整信息，增删改查直观
- 方便用Excel/pandas直接查看和编辑，运营人员也能操作
- label和sku_id作为两个独立变量，不需要像JSON那样嵌套层级

### 匹配逻辑：两级检索 + Ratio Test两步验证

```
查询图 → 模型提取384维特征 → L2归一化
                                    ↓
        ┌───────────────────────────────────────────────┐
        │  第一级：label级检索                            │
        │  query × sku_features.npy → 余弦相似度向量    │
        │  按 label 聚合，每 label 取最大相似度          │
        │  取 top5 labels                                │
        └───────────────────┬───────────────────────────┘
                            ↓
        ┌───────────────────────────────────────────────┐
        │  第二级：SKU级聚合 + Ratio Test               │
        │  top5 labels → 查 CSV 映射到 sku_id            │
        │  同一 sku_id 取最大相似度                       │
        │  取 top1 SKU 和 top2 SKU（第一个不同sku）      │
        │  计算 ratio = top1_sim / top2_sim              │
        │  若top5全为同一sku，则 ratio = ∞               │
        └───────────────────┬───────────────────────────┘
                            ↓
        ┌───────────────────────────────────────────────┐
        │  双重验证判定                                   │
        │  matched:  top1 ≥ 0.85 且 ratio ≥ 1.2       │
        │  low_conf: top1 ≥ 0.85 但 ratio < 1.2        │
        │  unmatched: top1 < 0.85                        │
        └───────────────────────────────────────────────┘
```

**Ratio Test两步验证详解**：
1. **第一步**：余弦相似度筛选 - top1相似度必须≥0.85，否则直接unmatched
2. **第二步**：Ratio验证 - 比较top1与top5中第一个不同sku的相似度比值
   - 如果top5 labels全属于同一sku，说明多个面都高度一致，ratio=∞，高置信matched
   - 如果top5中有不同sku，计算 ratio = top1_sim / first_different_sku_sim
   - ratio ≥ 1.2 表示top1显著高于其他候选，高置信matched
   - ratio < 1.2 表示候选之间差距不够大，可能存在混淆，需要人工复核(low_conf)

**为什么这样设计**：
1. **避免特征空间折叠**：训练时每个面是独立label，模型不需要把不同面强行拉近，特征空间保持良好的类间距离
2. **label级检索更精准**：先在更细粒度的label层找最相似的"面"，比直接在SKU层聚合噪声更小
3. **top5提供更多信息**：返回top5 labels让前端可以展示"疑似候选"，便于人工复核时参考
4. **Ratio Test两步验证兜底**：能自动识别"多个候选都很像"的模糊情况，将这类结果标记为low_conf而非直接匹配，降低误匹配风险

| YOLO输入 | numpy array | HWC格式, 640×640 |
| 裁剪图 | PIL.Image | 检测框区域, resize至224×224 |
| 特征向量 | torch.Tensor → numpy.ndarray | [384], float32 |
| 传输前端 | Base64字符串 | data:image/jpeg;base64,xxx |

### 6.3 SKU特征库数据结构

**关键原则**：SKU库特征和线上查询特征必须来自同一个模型，重新训练模型后必须重新建库。

**SKU库文件组成**：

```
sku_library/
├── sku_library.csv          # 图片-标签映射（核心索引文件）
├── sku_features.npy         # 特征矩阵 [N, 384], dtype=float32
├── images/                  # SKU图片存储（按SKU编号分目录）
│   ├── 000001/
│   │   ├── front.jpg
│   │   ├── aug_left_front.jpg
│   │   └── ...
│   ├── 000002/
│   └── ...
└── labels/                  # OML训练标签文件
```

**sku_library.csv 格式**：
```csv
image_name,sku_id,label,sku_name
000001/front.jpg,000001,000001_front,康师傅红烧牛肉面
000001/aug_left_front.jpg,000001,000001_front,康师傅红烧牛肉面
...
000001/side_a.jpg,000001,000001_side_a,康师傅红烧牛肉面
...
```

**sku_features.npy**：shape `[N_total_images, 384]`，dtype=float32，行顺序与CSV行顺序一致。

**特征来源说明**：
- **训练阶段**（离线，一次性）：OML Triplet Loss端到端微调ViT-S16模型，每个面作为独立label训练，训练产物是微调后的模型权重文件 `sku_vit_finetuned_best.pt`，此阶段不涉及npy特征文件
- **建库阶段**（离线，模型确定后执行）：用微调后的模型对SKU图片做前向推理，提取384维特征向量，L2归一化后存为 `sku_features.npy`，图片-标签映射存为 `sku_library.csv`
- **匹配阶段**（线上，每次请求）：查询裁剪图用同一个微调模型提取特征，和npy矩阵做余弦相似度，按label→SKU两级检索

## 7. 关键配置参数

### 7.1 检测参数

| 参数 | 默认值 | 说明 |

### 8.2 错误场景及用户体验设计

| 场景 | HTTP状态码 | 原因 | 用户看到的 | 恢复路径 |
|------|------------|------|------------|----------|
| 模型未加载 | 503 | 启动时模型加载失败或YOLO初始化异常 | 页面顶部红色横幅："⚠️ 系统初始化中，请稍后重试" + 重试按钮 | 点击重试按钮，或刷新页面 |
| 图片格式错误 | 400 | 上传文件不是有效的图片格式 | 上传区域红色边框 + 提示："请上传 JPG、PNG 格式的图片" | 选择正确格式的文件重新上传 |
| 检测无结果 | 200 | 图片中未检测到箱体 (count=0) | 结果区域显示空状态插画（箱货图标）+ "未检测到箱体，请确认图片中包含箱货" + 重新上传入口 | 点击"重新上传"按钮选择其他图片 |
| 匹配器未初始化 | 200 | SKU特征库文件不存在或加载失败 | 结果区域顶部黄色提示条："⚠️ 仅完成检测，SKU匹配功能未启用" + 检测框正常显示（灰色）+ 不显示匹配信息 | 检查 sku_library 目录配置，或联系管理员 |
| GPU不可用 | 200 | CUDA不可用，自动回退CPU | 正常运行（无特殊提示，后台自动切换） | 无需用户操作 |
| 请求超时 | 504 | 图片过大或服务器负载高 | 弹出提示："请求超时，请尝试更小的图片或稍后重试" | 点击确定后重新上传 |
| 服务器内部错误 | 500 | 代码异常或资源不足 | 页面显示错误页面："服务器内部错误，请稍后重试" | 刷新页面重试或联系管理员 |

### 8.3 关键错误场景详细说明

#### 场景一：模型未加载 (503)

**触发条件**：
- 启动时 `BoxDetector` 初始化失败
- 模型文件损坏或路径错误

**用户界面表现**：
```
┌─────────────────────────────────────────────────────────────┐
│ ⚠️ 系统初始化中，请稍后重试                    [重试]      │
└─────────────────────────────────────────────────────────────┘
  ↑
  红色背景，白色文字，固定在页面顶部
```

**技术处理**：
- 返回 HTTP 503 Service Unavailable
- 响应体：`{"detail": "Model not loaded, please retry"}`

#### 场景二：检测无结果 (200, count=0)

**触发条件**：
- 图片中不存在符合条件的箱体
- 置信度过滤后无结果

**用户界面表现**：
```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│                    [空状态插画：箱子图标]                     │
│                                                             │
│              未检测到箱体，请确认图片中包含箱货               │
│                                                             │
│                    [重新上传图片]                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**技术处理**：
- 返回 HTTP 200 OK
- 响应体：`{"success": true, "count": 0, "boxes": [], "matches": [], ...}`
- 前端检测到 count=0 时，隐藏统计卡片，显示空状态插画

#### 场景三：SKU库不存在，跳过匹配

**触发条件**：
- `sku_library/sku_features.npy` 文件不存在
- `sku_library/sku_library.csv` 文件不存在
- 特征库加载时解析错误

**用户界面表现**：
```
┌─────────────────────────────────────────────────────────────┐
│ ⚠️ 仅完成检测，SKU匹配功能未启用                              │
└─────────────────────────────────────────────────────────────┘
  ↑
  黄色提示条，显示在结果区域顶部
  检测框颜色全部为灰色（而非红/黄/绿）
  不显示 SKU 编号和相似度信息
```

**技术处理**：
- 返回 HTTP 200 OK
- 响应体包含检测结果，但 `matches` 字段为空或全部为 null
- 前端检测到匹配功能未启用时，显示黄色提示条，检测框显示灰色

## 9. 性能考虑

### 9.1 性能指标

> ⚠️ **说明**：以下性能数据为预期量级，实际性能受测试条件影响。如有实测数据，请标注具体测试环境。

| 项目 | 预期量级 | 影响因素 | 测试条件（待实测） |

| 模型大小 | ~5.8MB | 模型架构 | YOLO-cheap 轻量化设计 |

> **实测建议**：如需获得准确性能数据，建议在以下环境中测试：

### 9.2 优化方向

1. **模型预热**：启动时执行一次空推理，避免首次请求慢

4. **批量特征提取**：多个裁剪图一次batch推理

## 11. 已知局限与后续迭代

### 11.1 当前局限

| 局限 | 详细说明 | 影响 | 改进优先级 |
|------|----------|------|------------|
| **Box-Mask割裂** | YOLO-seg 同时输出 box 和 mask，但当前 SKU 匹配用 box 裁剪，mask 信息完全未参与。在遮挡场景下，box 裁剪会包含大量背景噪声，严重干扰特征提取。 | 这是当前系统精度最大的瓶颈。当箱体被部分遮挡时，背景区域占裁剪图比例升高，导致特征相似度下降，可能将正常品误判为 low_conf 或 unmatched。 | **P0 - 最高** |
| **闭集识别的硬边界** | 当前系统对库外 SKU 只能返回 Unknown，无法给出"最接近的候选"或"疑似新品"信号。在实际使用中，用户可能需要系统给出"这个箱体可能是新品"的提示，而非直接判定为无法匹配。 | 当出现新 SKU 时，系统无法主动提示用户"发现疑似新品"，需要人工主动发现并触发建库流程。 | **P2 - 中期** |
| **单图处理限制** | 当前只支持单张图片上传，不支持批量处理。在仓库盘点等需要处理大量图片时，用户需要逐张上传，效率较低。 | 批量盘点场景下，用户体验较差，需要反复操作。 | **P1 - 较高** |
| **遮挡增强的局限性** | 训练时的遮挡增强仅在box区域内随机叠加简单色块，与真实仓库中箱体堆叠产生的复杂遮挡效果有差距。 | 真实场景中的遮挡形态更加多样，模型可能无法泛化到所有遮挡模式。 | **P2 - 中期** |

### 11.2 后续迭代方向

| 方向 | 具体内容 | 预期收益 |
|------|----------|----------|
| **批量处理** | 支持多图上传、队列处理、进度显示 | 提升大规模盘点效率 |
| **新品发现提示** | 当 top1 相似度在 0.6-0.85 之间时，提示"疑似新品，建议建库" | 提升系统智能化程度 |
| **Mask引导裁剪** | 利用YOLO输出的mask信息，只裁剪前景区域，排除背景干扰 | 显著提升遮挡场景下的匹配精度 |
| **移动端适配** | 响应式布局优化，支持手机拍照直接上传 | 扩展使用场景 |
| **结果导出** | 支持将检测结果导出为Excel/CSV，方便与台账对比 | 提升盘点工作效率 |
